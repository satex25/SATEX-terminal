# v0.4.4 Post-Release Smoke Test

**Build under test:** `SATEX Setup 0.4.4.exe` (signed) at SHA-256 `<TBD-on-publish>`
**Run on:** Clean Windows 11 box (no prior SATEX install) AND on an upgrade box (v0.4.3 already installed)
**Estimated time:** 45–60 min single operator
**Pre-requisite:** Test Alpaca paper account credentials available (optional but required for Phase E)

## Pass criteria

Every checkbox below must be ticked **green** before the release is announced.
Any **RED** in a "Rollback trigger" row → yank the release per the procedure at
the bottom.

---

## Phase A · Artifact integrity (5 min)

Verify the bits on disk match what was published. Run BEFORE the installer
touches the system.

- [ ] **SHA-256 matches** the published checksum in the GH Release body / `checksums.txt`
  - Command: `certutil -hashfile "SATEX Setup 0.4.4.exe" SHA256`
- [ ] **File size sane** (60–110 MB; v0.4.3 was 88.46 MB — large deviation is suspicious)
- [ ] **Authenticode signature present**
  - Command: `signtool verify /pa /v "SATEX Setup 0.4.4.exe"`
  - Expected: `Successfully verified` with the CA chain printed
- [ ] **Signature subject matches** the CN field in `certs/satex-codesign.csr` (`SATEX Trading Systems`)
  - Manual: right-click → Properties → Digital Signatures → Details → View Certificate → Subject
- [ ] **Sign timestamp present** (sha256 RFC 3161 — required for long-term validity after cert expiry)

**Rollback trigger:** any of the four signature checks fails or the SHA mismatches.

First launch on a box that has never run SATEX.

- [ ] App window opens at 1920×1080 (or scales to display if smaller)

- [ ] **About modal** shows version `0.4.4` (Help → About, or whatever the menu path is)
- [ ] Vault directory created at the expected location:
  - With `SATEX_VAULT_ROOT` env var set → at that path
  - Without env var, walk-up for `.obsidian/` succeeds → at the repo root

**Rollback trigger:** crash within first 30s, missing Vault dir on a happy-path install, or CSP violations on a clean cold start.

## Phase D · v0.4.4 acceptance (15 min) — **the headline features**

### D1 · Sub-second per-symbol bucket preference

- [ ] **Settings modal opens** (gear icon or menu)
- [ ] **"Sub-second Candles · Crypto only" section visible** with rows for BTC and ETH
- [ ] Default selection is **250 ms** for both symbols (no pref persisted yet)
- [ ] Click `500 ms` for BTC → button visually swaps to `on` state
- [ ] Close Settings → reopen → BTC still shows `500 ms` (cache survives modal cycle)
- [ ] **Restart the app** → reopen Settings → BTC still shows `500 ms` (disk persistence works)
- [ ] On disk: `Vault/Settings/subsecond-prefs.md` exists and contains:
  ```json
  { "version": 1, "prefs": { "BTC": 500 } }
  ```
- [ ] Hand-edit the file to add `"NVDA": 250` → restart app → reopen Settings → NVDA entry is NOT present (sanitizer dropped non-crypto)

### D2 · Chart SUB badge

- [ ] Focus BTC → chart auto-snaps to `500 ms` timeframe (per the pref from D1)
- [ ] **SUB badge visible** in the chart legend reading `SUB · 500 ms`
- [ ] Badge color is **cyan** (`#00c8ff` via `--bb-accent`), NOT yellow/orange — visually distinct from any `SIM` badge present
- [ ] Hover badge → tooltip explains sub-second mode + crypto-only constraint
- [ ] Click `5s` timeframe → SUB badge **disappears**
- [ ] Click `250 ms` timeframe → SUB badge reappears as `SUB · 250 ms` (text updates)

### D3 · Cross-symbol behavior

- [ ] Switch focus from BTC → NVDA → chart leaves `500 ms` and lands on `1s` (auto-fallback)
- [ ] On NVDA, the `250 ms` and `500 ms` buttons are **disabled** (greyed)
- [ ] Hover the disabled button → tooltip says "Sub-second timeframes are crypto-only (BTC/ETH). Equity feeds (IEX) cap at 1-second snapshots."
- [ ] Switch back to BTC → chart auto-snaps to `500 ms` again (pref re-applies)
- [ ] Manually click `1s` on BTC mid-session → tf is `1s`, badge gone
- [ ] Switch to ETH → switch back to BTC → tf snaps to `500 ms` (auto-snap fires on symbol change, ignoring the in-session manual click)

### D4 · S1-9 Auto-update toast

- [ ] Toast does NOT appear on launch (no newer version published yet)
- [ ] DevTools console shows no `electron-updater` errors
- [ ] Manual: in DevTools, dispatch a synthetic `UPDATE_AVAILABLE` event → toast appears with `[Restart Now]` button **disabled**
- [ ] Synthetic `downloaded: true` → `[Restart Now]` button enables
- [ ] (Optional — only after a v0.4.5 is published) Real update flow end-to-end

**Rollback trigger:** any of D1's persistence steps fails, the SUB badge doesn't render under `showSub`, or the auto-snap clobbers a mid-session manual tf.

## Rollback procedure (if any rollback trigger fires)

1. **Hide the GitHub Release** immediately (`gh release edit v0.4.4 --draft`) — keeps the tag but unpublishes the binary
2. **Delete the tag locally and remotely** if the issue is in the code state itself:
   - `git tag -d v0.4.4`
   - `git push origin :refs/tags/v0.4.4`
3. **Pin a notice** in the GH Release notes: `⚠️ v0.4.4 yanked due to [specific issue]. Stay on v0.4.3 until v0.4.5 is published.`
4. **File a P0 issue** referencing the failing checklist item
5. **Notify** any analyst already running v0.4.4 (no auto-channel today — Sprint 3 will wire telemetry; until then, manual outreach)
6. **Auto-update path**: v0.4.4 had toast-gated updates with `autoDownload: false`, so a wide blast radius is unlikely; users who already accepted the update need a manual heads-up

## Notes for the operator

1. **Tooling assumed**: Windows native (`certutil`, `signtool`), no external test framework — runnable by one operator with a paper Alpaca account.
2. **The auto-update test (D4)** is best-effort until a v0.4.5 is actually published; the synthetic-event approach is the next best thing.
